## Arquitetura Geral

```
Lead clica em "Analisar meu Negócio"
       ↓
Chat horizontal em 8 etapas (slider)
       ↓
E-mail → Fase → Gargalo → URL do site
       ↓
Jina AI raspa o site (bypassa Cloudflare/JS)
       ↓
Gemini 2.5 Flash cruza: site + fase + gargalo + serviços ConnectIdeia
       ↓
Diagnóstico em 3 itens animados (negócio / gap / ação)
       ↓
Timing → WhatsApp → dados salvos no Supabase
```

## 2. Backend — `/api/analyze.js`

**Tecnologia:** Node.js Serverless (Vercel), ES Modules (`"type": "module"`)

**Fluxo:**
1. Recebe `{ url, phase, bottleneck }` do chat
2. Scraping via **Jina AI Reader** (`https://r.jina.ai/URL`) — bypassa Cloudflare, lida com JS, timeout 15s
3. Se falhar → retorna `{ skipped: true }` e chat avança silenciosamente
4. Gemini 2.5 Flash cruza conteúdo do site com respostas do chat
5. Retorna diagnóstico JSON:
```json
{
  "negocio": "O que o lead faz — identificação precisa",
  "gap": "Principal problema digital detectado no site",
  "acao": "Como a ConnectIdeia resolve com resultado tangível"
}
```

**Regras do prompt:**
- Máximo 2 linhas / 25 palavras por campo
- Diagnóstico cirúrgico, não relatório
- Dados reais do site — nunca inventar
- Ação deve mencionar resultado tangível (ex: "3x mais leads", "80% menos atendimento manual")
- Sem aspas dentro dos valores JSON

**Matriz de serviços fase × gargalo (3×4 = 12 combinações):**

A função `getService(phase, bottleneck)` cruza os dois inputs para retornar a recomendação mais cirúrgica:

| Fase \ Gargalo | LP | App | Automação | Ecossistema |
|---|---|---|---|---|
| **Ideia** | LP de validação de mercado | MVP em PWA para validar com usuários | Operar sem time grande via WhatsApp+CRM | Base digital do zero: LP+MVP+Auto |
| **Operação** | LP de alta conversão do tráfego atual | App para digitalizar processos sem aumentar time | Eliminar gargalos de atendimento/CRM | Integrar LP+App+Auto em stack coesa |
| **Escala** | LPs segmentadas para multiplicar captação | App como canal de receita próprio | Volume crescente sem custo proporcional | Máquina integrada de crescimento |

O Gemini recebe instrução explícita de que a fase muda o **tom e o ângulo** da `acao` (não é só contexto decorativo).

## 4. Frontend — Chat Conversacional (`index.html`)

### Estrutura
- Reutiliza o shell `pop-up="cta"` do Webflow (modal existente na LP)
- Slider horizontal com 8 etapas via CSS `transform: translateX`
- Wrapper com altura dinâmica via JS + `transition: height 0.45s`

### Ordem das Etapas (State Machine)

| Step | Tipo | Conteúdo |
|---|---|---|
| 0 | text/email | E-mail corporativo |
| 1 | options | Fase: Ideia / Operação / Escala |
| 2 | options | Gargalo: LP / App / Automação / Ecossistema |
| 3 | text/url | URL do site → dispara análise |
| 4 | analysis | Loading → 3 itens animados (fade-in sequencial) |
| 5 | options | Timing: urgência do projeto |
| 6 | text/tel | WhatsApp → salva no Supabase → step 7 |
| 7 | end | "✓ Recebido!" |

### Comportamento da Análise (Step 4)
- Loading spinner até resposta da API
- Conteúdo inserido no DOM antes de exibir (garante altura correta)
- Wrapper re-mede altura em 30ms após exibir resultados
- Items revelados com fade-in sequencial: 200ms / 700ms / 1300ms
- Botão "Faz sentido — continuar →" aparece em 1800ms

### Proteções UX
- `isProcessing` flag → previne double-submit e fechamento durante análise
- Validação de e-mail (regex) e WhatsApp (mínimo 8 dígitos) com feedback visual
- Modal não fecha durante processamento
- `type="button"` em todos os botões → previne submit do Webflow
- `onsubmit="return false;"` no form

### CTA
- Texto: **"Analisar meu Negócio"** com ícone de lupa (🔍 / SVG)
- Cor: branco com texto preto (botão pill)

## 6. Decisões Técnicas

| Decisão | Motivo |
|---|---|
| Jina AI Reader | Fetch direto bloqueado por Cloudflare e JS-rendered sites |
| Gemini 2.5 Flash | Único modelo disponível sem erro 404 na API key atual |
| `"type": "module"` | Vercel Node 18+ exige ES Modules para `import` |
| Diagnóstico JSON estruturado | Permite 3 balões/itens separados com animação individual |
| Slider horizontal vs chat de bolhas | Mais limpo, sem scroll interno, altura controlável |
| Reuse do shell Webflow `pop-up="cta"` | Evita conflito com o sistema de modal existente |
| Altura dinâmica JS | Cada step tem altura diferente; CSS transition suaviza |
| Conteúdo no DOM antes de exibir | Garante medição de altura correta antes de expandir o painel |

## 7. Batalha com o CSS do Webflow

O CSS externo do Webflow (CDN, minificado) aplica grid e estilos que conflitam com o chat. Overrides necessários:

```css
.pop-up_cta_form-block_form,
.pop-up_cta_c,
.pop-up_cta_modal { display: block !important; grid-template-columns: none !important; }

.ci-flow-track-wrapper { display: block !important; overflow: hidden !important; }
.ci-flow-track { display: flex !important; flex-wrap: nowrap !important; }
.ci-flow-step { min-width: 100% !important; flex-shrink: 0 !important; }
.ci-flow-step-inner { display: flex !important; }
.ci-flow-input-group { display: flex !important; flex-direction: row !important; flex-wrap: nowrap !important; }
.ci-flow-input-group > input { flex: 1 !important; min-width: 0 !important; }
.ci-flow-input-group > button { flex-shrink: 0 !important; position: static !important; display: flex !important; }
```

> `.ci-flow-loading` NÃO leva `!important` no display — o JS precisa sobrescrever com `style.display = 'none'`.

## 8. Twilio — Integração WhatsApp

**Endpoints:**
- `/api/whatsapp-send.js` — envia msg automática com diagnóstico quando cliente confirma WhatsApp
- `/api/whatsapp-webhook.js` — recebe msgs entrantes do cliente e salva no Supabase
- `/api/admin-login.js` — valida senha do admin

**Tabela Supabase:** `whatsapp_conversations`
```sql
create table whatsapp_conversations (
  id bigint generated by default as identity primary key,
  from_number text not null,
  message_body text not null,
  message_sid text unique,
  direction text check (direction in ('incoming', 'outgoing')),
  created_at timestamptz default now()
);
```

**Fluxo:**
1. Cliente entra WhatsApp em step 6
2. `/api/whatsapp-send` envia msg automática via Twilio com diagnóstico (negocio, gap, acao)
3. Twilio webhook envia para `/api/whatsapp-webhook` quando cliente responde
4. Mensagens armazenadas em `whatsapp_conversations`

**Webhook Twilio (Sandbox):**
- URL: `https://lp-connectideia.vercel.app/api/whatsapp-webhook`
- Method: POST

## 9. Admin Painel — `/api/admin.js`

**URL:** `https://lp-connectideia.vercel.app/api/admin`

**Funcionalidades:**
- Login por senha (`ADMIN_PASSWORD`)
- Filtros de leads: e-mail, fase, gargalo
- Grid de cards com e-mail, fase, gargalo, data e botão WhatsApp clicável (`wa.me/`)
- Cards clicáveis abrem `wa.me` direto no WhatsApp

**Estrutura:**
1. Valida cookie `admin_auth` — se ausente, mostra form de login
2. Login → `/api/admin-login` valida senha → seta cookie por 24h
3. Carrega leads via `/api/get-leads` (endpoint próprio, usa `SUPABASE_ANON_KEY`)
4. Filtros enviam `email`, `phase`, `bottleneck` como query params simples

**Endpoint `/api/get-leads.js`:**
- Busca leads na tabela `leads_chat` via Supabase client
- Suporta filtros: `email` (ilike), `phase` (eq), `bottleneck` (eq)
- Usa `SUPABASE_ANON_KEY` da env var (não hardcoded)

**Política RLS no Supabase (`leads_chat`):**
```sql
-- INSERT (leads do formulário)
create policy "Allow public inserts" on leads_chat for insert with check (true);
-- SELECT (admin panel)
create policy "Allow public selects" on leads_chat for select using (true);
```

> ⚠️ O painel admin não tem auto-refresh no momento — recarregar manualmente para ver novos leads.

## 12. Status do Twilio — Sandbox vs Produção

**Situação Atual (Sandbox):**
- Mensagens enviadas **do número sandbox** (`+14155238886`) do Twilio
- **No sandbox, só é possível enviar para números que fizeram o "join"** — ou seja, que enviaram o código de ativação para o número do Twilio
- Serve para testes internos; clientes reais **não recebem** mensagens no sandbox

**Custos do Twilio:**
- Sandbox: **gratuito** (sem limite, só para números verificados)
- Produção: **~US$1,15/mês** pelo número + **~R$0,12 por mensagem enviada**
- Exemplo: 100 leads = ~R$12 em mensagens

**Para Migrar para Produção:**
1. Comprar número no Twilio (~US$1,15/mês)
2. Solicitar aprovação para WhatsApp Business API via Twilio (processo Meta)
3. Atualizar env var no Vercel:
   ```bash
   vercel env rm TWILIO_WHATSAPP_FROM
   vercel env add TWILIO_WHATSAPP_FROM
   # Valor: whatsapp:+NÚMERO_COMPRADO
   ```
4. Deploy: `vercel --prod`

**Resultado após produção:**
- Qualquer cliente que preencher o formulário recebe a mensagem automaticamente
- Sem necessidade de "join" prévio
